# Контрольная работа №4

Данная папка содержит результаты выполнения практических заданий №19–23 по дисциплине «Фронтенд и бэкенд разработка». 

## Практическое занятие 19: Работа с PostgreSQL

**Цель:** Изучение основ работы с реляционными СУБД на примере PostgreSQL, включая настройку подключения и определение схем данных.

**Что было сделано:**
- Реализовано серверное приложение на Node.js с фреймворком Express.
- Настроено подключение к базе данных PostgreSQL с использованием библиотек `pg` и `sequelize`.
- Определены модели данных «Пользователь» (`User`) и «Задача» (`Task`), установлена связь «один-ко-многим».
- Реализован полный набор CRUD-операций (создание, чтение, обновление, удаление) для сущности «Пользователь».
- Создано полноценное REST API согласно заданию.


## Практическое занятие 20: Работа с MongoDB

**Цель:** Изучение принципов работы с документоориентированными NoSQL СУБД на примере MongoDB и библиотеки Mongoose.

**Что было сделано:**
- Установлена и настроена MongoDB, включая создание административного пользователя и включение аутентификации.
- Разработано серверное приложение с использованием `mongoose` для взаимодействия с базой данных.
- Описана Mongoose-схема для сущности «Пользователь» с полями `first_name`, `last_name`, `age`, `created_at`, `updated_at`.
- Реализованы все необходимые CRUD-эндпоинты для управления пользователями через REST API.


## Практическое занятие 21: Кэширование с Redis

**Цель:** Оптимизация производительности веб-приложения путем внедрения кэширования часто запрашиваемых данных с помощью Redis.

**Что было сделано:**
- В существующее приложение из практики №11 добавлен сервис кэширования Redis.
- Создано промежуточное ПО (`cacheMiddleware`) для автоматической проверки наличия данных в кэше перед обращением к основному источнику.
- Реализована функция `saveToCache` для сохранения ответов сервера в Redis с заданным временем жизни (TTL).
- Настроено кэширование для маршрутов:
    - `GET /api/users` (TTL: 1 минута)
    - `GET /api/users/:id` (TTL: 1 минута)
    - `GET /api/products` (TTL: 10 минут)
    - `GET /api/products/:id` (TTL: 10 минут)
- Добавлена логика инвалидации (сброса) кэша при обновлении или удалении данных, чтобы клиент всегда получал актуальную информацию.


## Практическое занятие 22: Балансировка нагрузки (Nginx/HAProxy)

**Цель:** Изучение и практическая настройка балансировки нагрузки для повышения отказоустойчивости и масштабируемости веб-приложений.

**Что было сделано:**
- Создано несколько экземпляров простого backend-сервера на Node.js, запущенных на разных портах (3000, 3001, 3002).
- Настроен Nginx в качестве балансировщика нагрузки:
    - Сконфигурирован блок `upstream` для распределения трафика.
    - Протестирован алгоритм балансировки по умолчанию (Round Robin).
    - Настроен резервный сервер (`backup`).
    - Добавлены параметры отказоустойчивости (`max_fails`, `fail_timeout`) для автоматического исключения неработающих узлов.
- Изучена и применена альтернативная конфигурация балансировки с помощью HAProxy.
- Настроена контейнеризация в Docker.


## Практическое занятие 23: Контейнеризация с Docker

**Цель:** Освоение технологии контейнеризации для упаковки приложений в изолированное окружение с помощью Docker и Docker Compose.

**Что было сделано:**
- Реализован собственный проект, включающий:
    - **Два идентичных backend-сервиса** на Node.js, возвращающих свой уникальный идентификатор.
    - **Nginx-балансировщик** в виде отдельного Docker-контейнера.
    - Файлы `Dockerfile` для сборки образов backend-сервисов.
    - Файл `docker-compose.yml`, описывающий все сервисы, их сетевые настройки и тома.
    - Конфигурационный файл `nginx.conf` с настройками балансировки и отказоустойчивости.
- Проведено тестирование: с помощью `curl` подтверждено поочередное распределение запросов между контейнерами и корректная работа `max_fails` при принудительной остановке одного из сервисов.
